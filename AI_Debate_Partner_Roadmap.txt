The Full VisionMost AI chatbots can argue back at you. That is not what this is. This system debates you, evaluates the quality of your reasoning in real time, detects logical fallacies in your arguments, adapts its difficulty to your performance, and produces a structured concept mastery report at the end — showing exactly where your understanding held up and where it collapsed.The system runs in three phases:Phase 1 — Debate Mode: AI takes an opposing position and debates you normally. This is where it collects data on your reasoning quality.Phase 2 — Probe Mode: AI switches from pure opposition to targeted questions designed to expose gaps. Difficulty adapts based on your performance.Phase 3 — Evaluation Mode: The system produces a structured mastery report — concept scores, gaps identified, fallacies committed, strongest and weakest arguments.Why This Is Resume-Worthy Across Every ML RoleML Engineer: argument parsing pipeline, classifier stack. AI Research: argument mining, evaluation methodology. EdTech / Product: intelligent tutoring system. Quant / Analyst: domain knowledge evaluation tool. This project hits all four simultaneously — which is rare.2. Full System ArchitectureThere are 7 distinct modules. Each one is independently buildable and testable, which is what makes the timeline realistic.ModuleWhat It DoesTech ApproachArgument ParserExtracts claim, premises, assumptions, evidence from raw inputGemini API + structured promptWeakness ScorerScores each premise on evidence, scope, and causality vulnerabilityGemini API + JSON rubric promptFallacy DetectorIdentifies logical fallacies in user's argumentGemini API + few-shot examplesAdaptive Question EngineGenerates harder questions targeting your weak nodesConcept graph (Python) + GeminiConsistency TrackerTracks AI and user positions, flags contradictionsPython dict + Gemini checkStance Firmness ModuleClassifies pushback as new argument, restatement, or emotionalFine-tuned DistilBERT classifierMastery EvaluatorAggregates all signals into a structured report at session endPython aggregation + Gemini narrative2.1 Module 1 — Argument ParserThis is the most important module. Every downstream component depends on its output quality. If the parser is sloppy, the entire system degrades.Input: raw user text. Output: structured JSON with the following fields:{  "main_claim": "Social media should be banned for under-16s",  "premises": [    "Social media causes depression in teenagers",    "Teenagers cannot handle social media responsibly"  ],  "implicit_assumptions": [    "Depression is primarily caused by social media, not other factors"  ],  "evidence_cited": [],  "confidence_language": "high"}The confidence_language field (high / medium / low) detects overconfident language — words like definitely, obviously, everyone knows. This is itself an evaluation signal: people who argue with high confidence but no evidence are a specific reasoning pattern worth flagging.Critical Implementation NoteSpend 30% of your total project effort getting this prompt exactly right with 5–6 few-shot examples. That investment compounds across every other module. A prompt that works on 9/10 inputs is the bar you need before moving on.2.2 Module 2 — Weakness ScorerTakes each premise from the parser output and scores it on three vulnerability dimensions:Evidence Score (0–1): Is this claim supported by data or just assertion? Pure assertions score 1.0 (maximally vulnerable).Scope Score (0–1): Is this claim overgeneralized? All teenagers vs. some teenagers scores much higher.Causal Score (0–1): Is the causal chain actually proven or just implied? Correlation presented as causation scores 1.0.The premise with the highest combined vulnerability score gets attacked first. This is what makes the AI feel like a skilled debater rather than a random counterargument generator.2.3 Module 3 — Fallacy DetectorChecks every user turn for these six fallacy types:FallacyWhat It CatchesExampleAd hominemAttacking the person not the argumentYou would say that, you're biasedFalse dichotomyPresenting only two options when more existEither we ban it or kids sufferStraw manMisrepresenting the AI's position to attack itSo you're saying kids should have no limits?Hasty generalizationBroad conclusion from limited examplesOne study proved it so it must be trueAppeal to authorityCiting authority without actual evidenceExperts agree, so it must be rightCircular reasoningUsing conclusion as a premiseIt's harmful because it causes harmCritical Warning — Over-DetectionLLMs love finding fallacies everywhere. Without careful few-shot examples and a confidence threshold, your system will flag every strong assertion as a fallacy. Test this explicitly: run a carefully written, logically clean argument through the detector. It should return zero fallacies. If it does not, your prompt needs more precision.2.4 Module 4 — Adaptive Question EngineThis is the most novel component and the one that elevates the project above everything else. It maintains a knowledge graph of the debate topic — key concepts, sub-concepts, and their relationships. As you argue, it tracks which concepts you have demonstrated understanding of and which you have avoided or handled weakly.Question generation logic:If you argued confidently about X but never addressed Y (which X depends on) → question targets YIf you made a claim that requires knowing Z to be defensible → question asks you to defend Z specificallyIf you committed a fallacy → next question probes whether it was intentional rhetoric or genuine misunderstandingStrong response → harder next question. Weak response → clarifying foundational question first2.5 Module 5 — Consistency TrackerMaintains a running log of every position the AI has taken and every position you have taken. Before generating any new response, checks for contradictions with prior statements. This makes the AI feel like a real opponent with memory — not a stateless chatbot.Implementation: a Python dict passed into every Gemini prompt as context. No database required. The LLM checks for contradiction when you include the history and explicitly instruct it to do so.2.6 Module 6 — Stance Firmness ClassifierThis solves the biggest problem with LLMs in debates: they cave the moment you push back, even when you have not made a new argument. The classifier categorizes your pushback into four types:Pushback TypeDefinitionAI ResponseNew argumentGenuinely new point not previously addressedEngage seriously, may adjust positionRestatementSame point rephrased more forcefullyHold firm, acknowledge but do not concedeEmotional pressureExpressing frustration or insistence without new logicAcknowledge feeling, maintain stanceConcession with reasoningAcknowledging a point while defending the overall positionHigh signal — score as strong reasoningThis is the only module that needs actual fine-tuning (vs. prompting). A DistilBERT classifier on Reddit CMV data handles this well and runs on CPU at inference.2.7 Module 7 — Mastery EvaluatorRuns at the end of the session. Aggregates all signals across all turns and produces the structured report. This is the money shot of the entire project.CONCEPT MASTERY REPORT━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━Topic: Universal Basic IncomeSession: 14 turns | 23 minutesOVERALL SCORE: 67/100STRENGTHS  ✓ Strong grasp of economic distribution arguments  ✓ Correctly cited empirical evidence in turns 4 and 9  ✓ Successfully defended against 3 of 5 probing questionsGAPS IDENTIFIED  ✗ Weak on inflationary effects of UBI (avoided twice)  ✗ No engagement with labor supply reduction counterargument  ✗ Assumed fiscal feasibility without defending itFALLACIES DETECTED  ⚠ Hasty generalization — Turn 6  ⚠ False dichotomy — Turn 11RECOMMENDED STUDY AREAS  → Monetary policy and inflation mechanics  → Labor economics fundamentals  → Government budget constraint theoryThe Test for a Good ReportTwo people who argued completely differently about the same topic should get completely different reports. The Core Gemini API CallEvery module — parser, scorer, fallacy detector, rebuttal generator — is just this function called with a different prompt. This is the backbone of your entire project.import google.generativeai as genaiimport jsongenai.configure(api_key="YOUR_API_KEY")model = genai.GenerativeModel("gemini-1.5-flash")def call_gemini(prompt: str) -> dict:    response = model.generate_content(prompt)    text = response.text.strip()    # Strip markdown code blocks if Gemini wraps in ```json    if text.startswith("```"):        text = text.split("```")[1]        if text.startswith("json"):            text = text[4:]    return json.loads(text)# Usage exampleresult = call_gemini(ARGUMENT_PARSER_PROMPT.format(user_input=user_text))# result is now a Python dict with claim, premises, assumptions, etc.Get your free API key from Google AI Studio (aistudio.google.com) — not Google Cloud. Possible Hurdles & How to Handle ThemHurdle 1 — Argument Parser Quality CascadeThe parser is the first step. If it extracts structure poorly, every downstream component degrades. A badly parsed argument feeds a bad weakness score feeds a bad rebuttal. This cascades fast.How to detect early: run 20 manually written arguments through it. Check if you would agree with the extracted structure if you read it cold.Target: 85%+ agreement before moving on to the next moduleHow to fix: add more diverse few-shot examples. The most common failure mode is the parser collapsing everything into one premise when there are actually three. Show it explicit examples of multi-premise arguments.Hurdle 2 — Fallacy Over-DetectionLLMs love finding fallacies everywhere. Without careful few-shot examples and a confidence threshold, your system will call every strong assertion an appeal to authority and every comparison a false analogy.How to detect: run a carefully written, logically clean argument through the detector. It should return zero fallacies. If it does not, your prompt is too aggressive.How to fix: add negative examples to your few-shot prompt — examples of strong assertions that are NOT fallacies. Explicitly instruct the model to only flag clear, unambiguous fallacies.Hurdle 3 — The Debate Feeling ScriptedIf the AI always attacks the same way in the same order, users figure out the pattern in 3 turns and the evaluation becomes gameable.How to detect: run the same argument 5 times. Responses should feel meaningfully different.How to fix: add a strategy selection step before rebuttal generation. The AI picks from 3 rebuttal types (counterevidence, scope reduction, causal challenge) based on the vulnerability dimension that scored highest. This injects variation at the structural level.Hurdle 4 — Adaptive Difficulty Not Actually AdaptingEasy to build a system that claims to adapt but actually just asks progressively longer questions. Real adaptation means the conceptual depth changes, not just surface complexity.How to detect: a domain expert should find early questions easy and later questions genuinely challenging. Test on yourself and someone who knows the topic well.How to fix: build the concept graph more carefully. Explicitly model which concepts are foundational vs. advanced within a topic. Early questions should target foundational nodes; advanced questions should target the relationships between nodes.Hurdle 5 — Mastery Report Feeling GenericYou should improve your understanding of the topic is useless. The report needs to cite specific turns, specific claims, specific gaps.How to detect: two people who argued completely differently on the same topic should get completely different reports.How to fix: pass the full structured session log into the evaluator prompt — not just a summary. Include every parsed argument JSON, every fallacy detected (with turn numbers), every question the user struggled with. The more raw material the evaluator has, the more specific the report.Hurdle 6 — Gemini JSON ReliabilityGemini occasionally wraps JSON output in markdown code fences or adds a preamble sentence. Your parser function needs to handle this gracefully.Use the call_gemini wrapper from Section 6 which strips code fences automaticallyWrap every JSON parse in a try/except that retries the API call with a cleaner prompt if parsing failsAdd explicit instruction at the end of every prompt: Return ONLY valid JSON. No preamble, no explanation, no markdown formatting.9. The Kill Checklist — Am I On Track or Dead?Run this checklist at the end of each week.
